# neuro_and_iot_project

Веб-система для мониторинга психофизиологического состояния человека по данным с Arduino/ESP8266. Проект собирает показания датчиков, принимает их через MQTT или HTTP, отображает в браузере в реальном времени и оценивает состояние с помощью ML-модели.

Проект сейчас умеет различать три состояния:

- `calm` - спокойствие;
- `cognitive_load` - когнитивная нагрузка;
- `stressed` - стресс.

Оценка не является медицинским диагнозом. Это учебно-исследовательский прототип для анализа физиологических сигналов и демонстрации связки IoT + backend + ML + dashboard.

## Что уже сделано

В проекте реализован полный цикл от датчиков до веб-интерфейса:

- Arduino/ESP8266 считывает данные с датчиков MAX30102, DS18B20 и FSR402.
- Данные собираются в единый JSON и публикуются в MQTT-топик `sensor/health/data`.
- Backend на FastAPI подписывается на MQTT, принимает показания, валидирует их через Pydantic и передает в анализатор.
- Также есть HTTP POST-эндпоинт, чтобы отправлять данные напрямую без MQTT.
- Реализован встроенный генератор мок-данных, чтобы проверять систему без физического устройства.
- Реализовано скользящее окно измерений для расчета признаков.
- Реализована ML-модель, которая обучается на синтетических данных и может дообучаться на записанных реальных сессиях.
- Реализована персональная калибровка базового состояния, то есть baseline в покое.
- Реализовано сглаживание предсказаний, чтобы интерфейс не дергался от единичных скачков.
- Реализована запись размеченных сессий в JSON-файлы.
- Реализован веб-dashboard с графиками, текущими показаниями, стресс-индексом, уверенностью модели и важностью признаков.
- Реализован SSE-поток, через который браузер получает обновления в реальном времени.

## Архитектура

Общий поток данных выглядит так:

```text
Датчики
  -> ESP8266 / Arduino sketch
  -> MQTT broker
  -> FastAPI backend
  -> Sliding window + feature extraction
  -> ML model + smoothing
  -> SSE stream
  -> Browser dashboard
```

Альтернативный режим:

```text
HTTP POST /api/readings
  -> FastAPI backend
  -> ML analysis
  -> Dashboard
```

Еще один режим для разработки:

```text
Built-in mock generator
  -> FastAPI backend
  -> ML analysis
  -> Dashboard
```

## Аппаратная часть

В Arduino-скетче используется ESP8266/NodeMCU и три источника данных:

| Компонент | Назначение | Подключение в коде |
| --- | --- | --- |
| MAX30102 | IR-сигнал и расчет ЧСС | I2C, `SDA = D2`, `SCL = D1`, адрес `0x57` |
| DS18B20 | Температура | OneWire, пин `D5` |
| FSR402 | Датчик давления/сжатия | Аналоговый вход `A0` |

Скетч делает следующее:

- подключается к Wi-Fi;
- подключается к MQTT-брокеру;
- проверяет наличие MAX30102 и автоматически пытается переподключить его при потере;
- определяет наличие пальца по IR-порогу;
- рассчитывает текущий BPM и усредненный BPM;
- считывает температуру без блокирующего ожидания конверсии;
- усредняет 10 измерений FSR для более стабильного значения;
- публикует один общий JSON-снимок примерно раз в 3 секунды;
- сразу публикует первое валидное значение BPM, чтобы dashboard быстрее оживал;
- отправляет MQTT Last Will с состоянием `offline`.

Перед прошивкой нужно указать свои сетевые настройки в [arduino/arduino.ino](arduino/arduino.ino):

```cpp
const char* WIFI_SSID = "your_wifi";
const char* WIFI_PASS = "your_password";

const char* MQTT_HOST = "192.168.1.10";
const uint16_t MQTT_PORT = 1883;
```

Топик уже совпадает с backend:

```cpp
const char* TOPIC_DATA = "sensor/health/data";
```

Для Arduino IDE нужны библиотеки:

- ESP8266 board package;
- PubSubClient;
- OneWire;
- DallasTemperature;
- SparkFun MAX3010x Sensor Library, которая дает `MAX30105.h` и `heartRate.h`.

## Backend

Backend находится в [main.py](main.py). Он запускает FastAPI-приложение на `0.0.0.0:8000`.

Основные задачи backend:

- загрузить готовую модель из `models/stress_model.joblib`;
- если модель не найдена или версия не совпадает, обучить новую модель на синтетических данных;
- запустить MQTT-клиент в отдельном daemon-потоке;
- принимать сообщения из `sensor/health/data`;
- принимать ручные HTTP-показания через `/api/readings`;
- обрабатывать каждое измерение единым способом независимо от источника;
- обновлять скользящее окно;
- считать оценку состояния;
- рассылать результат всем открытым dashboard-клиентам через `/api/stream`;
- управлять baseline, сессиями, mock-генератором и переобучением модели.

MQTT-настройки backend:

```text
broker: localhost
port:   1883
topic:  sensor/health/data
```

Адрес брокера можно переопределить переменными окружения:

```powershell
$env:MQTT_BROKER="172.20.10.2"
$env:MQTT_PORT="1883"
python main.py
```

## Формат данных от Arduino

Arduino публикует один JSON-объект в топик `sensor/health/data`.

Пример:

```json
{
  "device": "online",
  "state": "measuring",
  "max30102_ready": true,
  "finger": true,
  "warmup": false,
  "ir": 45000,
  "bpm_valid": true,
  "bpm_current": 72.5,
  "bpm_avg": 72,
  "bpm_samples": 4,
  "temp_valid": true,
  "temp_c": 36.6,
  "fsr_raw": 160,
  "fsr_pressed": true
}
```

Важные поля:

| Поле | Что значит |
| --- | --- |
| `device` | состояние устройства, например `online` или `offline` |
| `state` | состояние измерения: `measuring`, `no_finger`, `sensor_missing`, `offline` |
| `max30102_ready` | найден ли датчик MAX30102 |
| `finger` | обнаружен ли палец |
| `warmup` | короткий прогрев после повторного захвата пальца |
| `ir` | текущее IR-значение MAX30102 |
| `bpm_valid` | можно ли доверять текущему пульсу |
| `bpm_current` | текущий пульс |
| `bpm_avg` | усредненный пульс по короткому буферу |
| `bpm_samples` | количество валидных BPM-сэмплов в буфере |
| `temp_valid` | валидна ли температура |
| `temp_c` | температура в градусах Цельсия |
| `fsr_raw` | сырое значение давления с FSR |
| `fsr_pressed` | превышен ли порог нажатия FSR |

Backend добавляет измерение в окно только если:

- `state == "measuring"`;
- `bpm_valid == true`.

Если этих условий нет, dashboard показывает `НЕТ СИГНАЛА`, а сглаживатель предсказаний сбрасывается.

## Анализ и ML-модель

ML-часть находится в папке [analyzer](analyzer).

### Скользящее окно

В [analyzer/window.py](analyzer/window.py) реализовано окно измерений:

- максимальная длина окна: `25` измерений;
- первый прогноз можно получить после `8` валидных измерений;
- при Arduino-интервале 3 секунды полное окно покрывает примерно 75 секунд.

Пока данных меньше 8, dashboard показывает состояние `РАЗОГРЕВ`.

### Признаки

В [analyzer/features.py](analyzer/features.py) извлекаются 12 признаков:

- средний пульс;
- вариация пульса;
- размах пульса;
- тренд пульса;
- средняя температура;
- вариация температуры;
- тренд температуры;
- среднее давление;
- вариация давления;
- отклонение пульса от baseline;
- отклонение температуры от baseline;
- отклонение давления от baseline.

Если baseline еще не откалиброван, относительные признаки считаются нулевыми.

### Модель

В [analyzer/model.py](analyzer/model.py) реализован класс `StressModel`.

Модель сравнивает два алгоритма:

- `RandomForestClassifier`;
- `GradientBoostingClassifier`.

При обучении backend:

- масштабирует признаки через `StandardScaler`;
- считает `cross_val_score` с `cv=5`;
- сохраняет метрики обеих моделей;
- выбирает модель с лучшей средней accuracy;
- сохраняет результат в `models/stress_model.joblib`;
- хранит важности признаков для объяснения результата.

Текущая версия модели: `MODEL_VERSION = "3"`. Если версия сохраненного файла отличается, модель переобучается.

### Синтетические данные

В [analyzer/synthetic.py](analyzer/synthetic.py) создается синтетический датасет:

- 600 окон на каждый класс;
- классы: `calm`, `cognitive_load`, `stressed`;
- одно окно содержит 20 измерений;
- данные генерируются с разными средними значениями пульса, температуры и FSR.

Это позволяет запускать проект сразу, даже если реальных сессий еще нет.

### Дообучение на реальных сессиях

Записанные сессии можно использовать для переобучения модели. Backend берет реальные сессии из папки `sessions`, режет их на окна по 20 измерений с перекрытием 50 процентов и добавляет к синтетическому датасету.

Реальные окна усиливаются весом `5:1`, то есть повторяются 5 раз при обучении:

```python
X = np.vstack([syn_X, np.repeat(real_X, 5, axis=0)])
y = np.concatenate([syn_y, np.tile(real_y, 5)])
```

Переобучение запускается из вкладки `Модель` или через API:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/model/retrain -Method Post
```

## Baseline-калибровка

Baseline нужен, чтобы модель могла учитывать индивидуальную норму человека.

Реализовано в [analyzer/baseline.py](analyzer/baseline.py):

- пользователь запускает запись baseline в состоянии покоя;
- backend собирает минимум 20 валидных измерений;
- при интервале Arduino 3 секунды это примерно 60 секунд;
- после остановки считаются средний BPM, разброс BPM, средняя температура и среднее давление FSR.

После калибровки признаки `bpm_z`, `temp_delta` и `fsr_delta` начинают отражать отклонение от личного спокойного состояния.

## Сглаживание предсказаний

В [analyzer/smoother.py](analyzer/smoother.py) реализовано сглаживание последних 5 прогнозов:

- итоговая метка выбирается большинством голосов;
- стресс-индекс усредняется;
- уверенность модели усредняется.

Это сделано, чтобы состояние на dashboard менялось стабильнее и не прыгало из-за одного шумного измерения.

## Веб-интерфейс

Dashboard находится в [static/index.html](static/index.html). Он открывается по адресу:

```text
http://127.0.0.1:8000/
```

Если backend запущен на компьютере в локальной сети, dashboard можно открыть с другого устройства по IP компьютера:

```text
http://192.168.1.10:8000/
```

В интерфейсе есть три вкладки.

### Монитор

Показывает:

- состояние подключения;
- текущую оценку: `СПОКОЙСТВИЕ`, `НАГРУЗКА`, `СТРЕСС`, `РАЗОГРЕВ` или `НЕТ СИГНАЛА`;
- стресс-индекс в процентах;
- уверенность модели;
- текущий BPM;
- температуру;
- давление FSR;
- графики BPM, температуры, FSR и стресс-индекса;
- ключевые факторы, которые сильнее всего влияют на модель;
- управление baseline;
- встроенный генератор данных.

### Сессии

Позволяет записывать размеченные данные:

- `Спокойствие`;
- `Когнитивная нагрузка`;
- `Стресс`.

После остановки сессия сохраняется в папку `sessions` в JSON-формате. Эти данные потом используются для переобучения модели.

### Модель

Показывает:

- метрики обученных моделей;
- активную выбранную модель;
- важность признаков;
- кнопку переобучения на синтетических и записанных данных.

Интерфейс использует Tailwind CSS и Chart.js через CDN, поэтому для полноценного отображения графиков и стилей нужен доступ к этим CDN или заранее закешированные ресурсы.

## API

Основные endpoint-ы:

| Метод | Путь | Назначение |
| --- | --- | --- |
| `GET` | `/` | открыть dashboard |
| `POST` | `/api/readings` | принять одно измерение |
| `GET` | `/api/stream` | SSE-поток обновлений для dashboard |
| `POST` | `/api/baseline/start` | начать запись baseline |
| `POST` | `/api/baseline/stop` | завершить baseline и посчитать статистику |
| `GET` | `/api/baseline/status` | получить состояние baseline |
| `POST` | `/api/session/start` | начать запись размеченной сессии |
| `POST` | `/api/session/stop` | остановить и сохранить сессию |
| `GET` | `/api/session/status` | получить статус текущей сессии |
| `GET` | `/api/sessions` | получить список сохраненных сессий |
| `GET` | `/api/mock/status` | проверить состояние mock-генератора |
| `POST` | `/api/mock/start` | запустить mock-генератор |
| `POST` | `/api/mock/stop` | остановить mock-генератор |
| `GET` | `/api/model/status` | получить состояние модели и окна |
| `POST` | `/api/model/retrain` | переобучить модель |
| `GET` | `/api/model/explain` | получить важности признаков |
| `POST` | `/api/window/reset` | очистить окно измерений |

Пример ручной отправки измерения без MQTT:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/readings `
  -Method Post `
  -ContentType "application/json" `
  -Body '{"device":"test","state":"measuring","bpm_valid":true,"bpm_current":72.5,"temp_valid":true,"temp_c":36.6,"fsr_raw":160}'
```

Проверка состояния модели:

```powershell
Invoke-RestMethod http://127.0.0.1:8000/api/model/status
```

Если данные доходят, поле `window_size` будет расти.

## Запуск проекта

Из корня проекта:

```powershell
python -m pip install -r requirements.txt
python main.py
```

После запуска открой:

```text
http://127.0.0.1:8000/
```

При запуске backend:

1. загружает или обучает модель;
2. стартует FastAPI-сервер;
3. запускает MQTT-подписку в отдельном потоке;
4. начинает ждать данные от Arduino, HTTP или mock-генератора.

## Проверка без Arduino

Самый быстрый способ проверить проект без железа:

1. Запустить backend:

```powershell
python main.py
```

2. Открыть dashboard:

```text
http://127.0.0.1:8000/
```

3. На вкладке `Монитор` включить `Генератор данных`.

Доступные сценарии:

- `Спокойствие`;
- `Когнитивная нагрузка`;
- `Стресс`.

Интервал генератора по умолчанию: 1 секунда. Его можно изменить через переменную окружения:

```powershell
$env:MOCK_INTERVAL="0.5"
python main.py
```

## Работа через MQTT

Для режима с реальным устройством нужен MQTT-брокер, доступный и компьютеру с backend, и ESP8266.

Backend по умолчанию подключается к:

```text
localhost:1883
```

Если брокер на другом устройстве:

```powershell
$env:MQTT_BROKER="192.168.1.10"
$env:MQTT_PORT="1883"
python main.py
```

В Arduino-скетче должен быть указан тот же IP:

```cpp
const char* MQTT_HOST = "192.168.1.10";
const uint16_t MQTT_PORT = 1883;
```

## Что сохраняется на диск

Проект сохраняет два типа данных:

- обученную модель в `models/stress_model.joblib`;
- записанные сессии в `sessions/*.json`.

Сессия содержит:

- ID;
- метку состояния;
- время старта и остановки;
- список измерений;
- рассчитанный на тот момент `stress_score`.
